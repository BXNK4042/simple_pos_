# การปรับปรุงเวิร์กโฟลว์เครื่องสแกนบาร์โค้ด: การดักจับสินค้าที่ไม่มีในระบบ (Unknown Product Interception)

ยินดีต้อนรับสู่ทีมของเราครับ! 🎉 หนึ่งในภารกิจแรกของคุณคือการช่วยพวกเราปรับปรุงประสบการณ์การชำระเงินที่หน้าเคาน์เตอร์แคชเชียร์ และช่วยยกระดับคุณภาพของข้อมูลในระบบให้ดีขึ้น 🛒

ในปัจจุบัน เมื่อแคชเชียร์หรือตัวเครื่องสแกนบาร์โค้ดส่งข้อมูลบาร์โค้ดที่ไม่รู้จักเข้ามา ระบบหลังบ้าน (Backend) จะสร้างสินค้าจำลอง (Placeholder) ขึ้นมาในฐานข้อมูลโดยอัตโนมัติ โดยตั้งชื่อว่า `"Unknown <barcode>"` พร้อมกำหนดราคาเป็น `0` และสต็อกเป็น `0` จากนั้นสินค้าชิ้นนี้จะถูกเพิ่มเข้าไปในตะกร้าปัจจุบันทันที

แม้ว่าวิธีนี้จะช่วยให้ระบบเวอร์ชันแรก (MVP) ทำงานได้ง่ายไม่ซับซ้อน แต่ก็นำมาซึ่งปัญหาหลายอย่างบนระบบจริง (Production):

1. **ข้อมูลขยะ (Data Pollution)**: ฐานข้อมูลเต็มไปด้วยสินค้าจำลองที่ยังไม่ได้ตั้งค่าข้อมูลจริง
2. **สูญเสียรายได้ (Loss of Revenue)**: สินค้าราคา 0 บาทอาจจะหลุดขายออกไปให้กับลูกค้าได้โดยไม่ตั้งใจ
3. **สต็อกผิดพลาด (Inventory Errors)**: สินค้าจำลองเหล่านี้ไม่มีการนับสต็อกจริง ไม่มีหมวดหมู่ และไม่มีการบันทึกข้อมูลต้นทุน

เพื่อแก้ไขปัญหานี้ เราต้องการดักจับบาร์โค้ดที่ไม่รู้จัก และแสดงหน้าต่างแจ้งเตือน (Warning Popup) ให้แคชเชียร์ทราบ หากแคชเชียร์กดยืนยัน พวกเขาจะสามารถลงทะเบียนสินค้าใหม่ได้ทันทีผ่านกล่องโต้ตอบ (Dialog) ที่มีอยู่เดิม ซึ่งเมื่อกดบันทึกแล้ว ระบบจะเพิ่มสินค้านั้นเข้าตะกร้าให้โดยอัตโนมัติครับ! 💡

## วัตถุประสงค์ (Objectives)

เป้าหมายของคุณคือการเปลี่ยนวิธีจัดการกับบาร์โค้ดที่ไม่รู้จัก แทนที่จะปล่อยให้ระบบหลังบ้านสร้างสินค้าจำลองขึ้นมาเงียบ ๆ ให้เปลี่ยนมาทำตามขั้นตอนต่อไปนี้แทน:

1. **ดักจับการสแกนที่ไม่รู้จัก (Intercept Unknown Scans)**: ส่งสัญญาณเฉพาะ (Distinct Signal) กลับมาจาก API เมื่อไม่พบสินค้าในระบบ
2. **แจ้งเตือนแคชเชียร์ (Warn the Cashier)**: แสดงกล่องโต้ตอบยืนยัน (Confirmation Dialog) เพื่อเตือนแคชเชียร์ว่าไอเทมที่สแกนไม่มีอยู่ในฐานข้อมูล
3. **ลงทะเบียนสินค้า (Register the Product)**: หากแคชเชียร์กดยืนยัน ให้เปิด NewProductDialog โดยกรอกบาร์โค้ดที่สแกนให้โดยอัตโนมัติ
4. **เพิ่มเข้าตะกร้าอัตโนมัติ (Auto-Add to Cart)**: หลังจากสร้างสินค้าสำเร็จแล้ว ให้เพิ่มสินค้านั้นเข้าตะกร้าโดยอัตโนมัติพร้อมราคาและรายละเอียดที่ถูกต้อง
5. **ปรับปรุงสิทธิ์การใช้งาน (Adjust Permissions)**: ตรวจสอบให้แน่ใจว่าผู้ใช้ระดับแคชเชียร์ (Cashier) ได้รับสิทธิ์ในการสร้างสินค้าผ่านเวิร์กโฟลว์นี้ได้ (ซึ่งในปัจจุบันถูกจำกัดไว้ให้เฉพาะเจ้าของร้าน หรือ Owner เท่านั้น)

### 1. แผนผังโครงสร้างซอร์สโค้ด (Codebase Architecture Map)

ก่อนจะเริ่มลงมือเขียนโค้ด มาทำความคุ้นเคยกับไฟล์สำคัญเหล่านี้กันก่อนครับ:

* **API Router สำหรับการสแกน**: app/api/scan/route.ts
*ทำหน้าที่จัดการทั้งการสแกนจากเครื่องสแกนและจากที่แคชเชียร์พิมพ์ค้นหาเข้ามาเอง ปัจจุบันโค้ดส่วนนี้เป็นคนสร้างสินค้าจำลองขึ้นมา*
* **SSE Scan Stream**: app/api/scans/stream/route.ts
*ทำหน้าที่รักษาการเชื่อมต่อแบบ Server-Sent Events (SSE) เพื่อใช้ส่งข้อมูลการสแกนไปยังหน้าจอแคชเชียร์*
* **ตัวส่งเหตุการณ์สแกน (Scan Event Emitter)**: lib/scan-events.ts
*กำหนดโครงสร้างข้อมูล (Shape) ของ `ScanResult` และจัดการตัวรับฟังก์ชัน (Listeners) ภายในหน่วยความจำ*
* **คอมโพเนนต์หน้าแคชเชียร์ (Cashier Page Component)**: components/cashier-pos.tsx
*หน้าแดชบอร์ด POS ของแคชเชียร์ ทำหน้าที่คอยฟังเหตุการณ์การสแกนจาก SSE และจัดการการส่งข้อมูลบาร์โค้ดแบบป้อนเอง*
* **กล่องโต้ตอบสินค้าใหม่ (New Product Dialog)**: components/new-product-dialog.tsx
*คอมโพเนนต์กล่องโต้ตอบที่มีอยู่แล้ว สำหรับใช้ในการสร้างสินค้าใหม่*
* **API สำหรับสินค้า (Products API)**: app/api/products/route.ts
*จัดการการค้นหาสินค้า (`GET`) และการสร้างสินค้า (`POST`)*

### 2. งานที่ได้รับมอบหมาย (Guided Tasks)

#### งานที่ 1: แก้ไข Scan API

ปัจจุบันในไฟล์ `app/api/scan/route.ts` จะรันโค้ดลักษณะนี้อยู่:

```typescript
if (!product) {
  product = await prisma.product.create({ ... })
}

```

* **สิ่งที่คุณต้องทำ**: เปลี่ยนตรรกะ (Logic) ตรงนี้ หากไม่พบสินค้า **ห้าม** สร้างสินค้าขึ้นมาใหม่ แต่ให้ส่งกลับหรือ Emit การตอบกลับในสถานะ "unknown" แทน
* **คำแนะนำ**: ลองนึกถึงข้อตกลงของข้อมูล (Data Contract) ดูครับ หากคุณเปลี่ยนโครงสร้างของ `ScanResult` ในไฟล์ `lib/scan-events.ts` ให้มีแฟล็กบอกสถานะเพิ่มเข้ามา (เช่น `status: "ok" | "unknown"`) หน้าแคชเชียร์ก็จะรู้ได้ทันทีว่าควรจะตอบสนองอย่างไร

#### งานที่ 2: อัปเดตเหตุการณ์และประเภทข้อมูลการสแกน (Scan Events & Types)

ถ้าเครื่องสแกนสแกนบาร์โค้ดที่ไม่รู้จัก หน้าจอแคชเชียร์จะต้องได้รับเหตุการณ์นั้นผ่านสตรีม Server-Sent Events (SSE)

* **สิ่งที่คุณต้องทำ**: แก้ไข `ScanResult` ใน lib/scan-events.ts เพื่อให้รองรับการเก็บข้อมูลบาร์โค้ดที่ไม่รู้จักและสถานะ (Status) แบบอัปชันนอล (Optional)
* **คำแนะนำ**: ตรวจสอบให้แน่ใจว่าใน `app/api/scan/route.ts` ยังคงเรียกใช้ `emitScan` สำหรับการสแกนจากเครื่องสแกนอยู่ แม้ว่าสินค้านั้นจะไม่รู้จักก็ตาม เพื่อให้ SSE listener ฝั่งหน้าจอแคชเชียร์สามารถดักจับเหตุการณ์นี้ได้

#### งานที่ 3: ออกแบบขั้นตอนกล่องโต้ตอบในฝั่งหน้าบ้าน (Frontend Dialog Flow)

ในไฟล์ components/cashier-pos.tsx คุณต้องดักจับการสแกนที่ไม่รู้จักให้ได้:

1. **การสแกนจากเครื่องสแกน (Hardware Scans)**: ดักจับเหตุการณ์จากฟังก์ชันรับข้อความ `EventSource` ของสตรีม SSE
2. **การสแกนแบบป้อนเอง (Manual Scans)**: ดักจับค่าที่ส่งกลับมาจากฟังก์ชัน `handleBarcodeSubmit`

* **สิ่งที่คุณต้องทำ**:
* สร้างตัวแปรสถานะ (State Variables) เพื่อใช้ควบคุมการเปิด/ปิดกล่องโต้ตอบแจ้งเตือนยืนยัน (เช่น ข้อความเขียนว่า "ไม่พบสินค้า คุณต้องการเพิ่มบาร์โค้ด `885...` เข้าไปในระบบหรือไม่?")
* หากกดยืนยัน ให้แสดง `NewProductDialog` พร้อมใส่ข้อมูลบาร์โค้ดลงไปในฟิลด์ให้เรียบร้อย


* **คำแนะนำ**: คุณสามารถสร้างคอมโพเนนต์ Dialog ขึ้นมาเอง หรือจะอิมพอร์ตคอมโพเนนต์ `Dialog` ของ shadcn จากไฟล์ components/ui/dialog.tsx เพื่อให้หน้าตา UI ดูสวยงามระดับพรีเมียมก็ได้ครับ 🎨

#### งานที่ 4: จัดการเรื่องสิทธิ์การใช้งานของบทบาท (Role Permissions)

หากคุณตรวจสอบในไฟล์ `app/api/products/route.ts` จะพบโค้ดส่วนนี้:

```typescript
export async function POST(request: Request) {
  const auth = await requireOwnerResponse()
  ...
}

```

โค้ดนี้จะจำกัดให้สิทธิ์ในการสร้างสินค้าเฉพาะผู้ใช้ระดับเจ้าของร้าน (Owner) เท่านั้น แต่หน้าจอแคชเชียร์นั้นถูกใช้งานโดยแคชเชียร์!

* **สิ่งที่คุณต้องทำ**: ตัดสินใจเลือกวิธีจัดการเรื่องนี้อย่างปลอดภัย หากแคชเชียร์สแกนเจอสินค้าใหม่ พวกเขาควรจะสามารถกดสร้างมันขึ้นมาได้
* **คำแนะนำ**: ลองดูที่ไฟล์ `lib/auth.ts` คุณสามารถเปลี่ยนจาก `requireOwnerResponse` เป็น `requireSessionResponse` ภายในตัวจัดการ (Handler) ของ `POST` ในไฟล์ app/api/products/route.ts ได้หรือไม่? และมีประเด็นเรื่องความปลอดภัย (Security trade-offs) อะไรบ้างที่ต้องเขียนบันทึกไว้ในเอกสาร?

#### งานที่ 5: เพิ่มสินค้าที่สร้างใหม่เข้าตะกร้าโดยอัตโนมัติ

เมื่อ NewProductDialog ทำการสร้างสินค้าสำเร็จ มันจะเรียกฟังก์ชัน `onCreated(product: NewProduct)` ขึ้นมาทำงาน

* **สิ่งที่คุณต้องทำ**: เขียนโค้ดในฟังก์ชันคอลแบ็ก (Callback) ของ `onCreated` ในไฟล์ components/cashier-pos.tsx โดยทำการแปลงข้อมูล `NewProduct` ที่ได้รับกลับมาให้อยู่ในรูปแบบที่รองรับกับฟังก์ชัน `cart.addItem` แล้วทำการเพิ่มสินค้านั้นเข้าไปในตะกร้าได้เลยครับ

### 3. การตรวจสอบและการทดสอบ (Verification & Testing)

ทดสอบการเปลี่ยนแปลงของคุณด้วยตัวเองผ่านสถานการณ์จำลองเหล่านี้ได้เลยครับ:

#### สถานการณ์ทดสอบ A: การสแกนแบบป้อนเอง (แคชเชียร์พิมพ์ข้อมูล)

1. เข้าสู่ระบบแล้วไปยังหน้า `/cashier`
2. พิมพ์บาร์โค้ดแบบสุ่มมั่ว ๆ (เช่น `99999999`) ลงในช่องค้นหาแล้วกด Enter
3. **ผลลัพธ์ที่คาดหวัง**: จะต้องมีกล่องโต้ตอบแจ้งเตือนเด้งขึ้นมา แทนที่จะมีสินค้าชื่อ `"Unknown 99999999"` โผล่เข้าไปในตะกร้าสินค้า
4. คลิก **Cancel (ยกเลิก)** -> กล่องโต้ตอบจะปิดลง และตะกร้าสินค้าต้องยังคงว่างเปล่าเหมือนเดิม
5. ค้นหา `99999999` อีกครั้ง -> กล่องโต้ตอบแจ้งเตือนเด้งขึ้นมา -> คลิก **Confirm (ยืนยัน)**
6. **ผลลัพธ์ที่คาดหวัง**: กล่องโต้ตอบสำหรับสร้าง "สินค้าใหม่" จะเปิดขึ้นมาพร้อมกรอกบาร์โค้ด `99999999` ให้ในช่องเรียบร้อย
7. กรอกชื่อสินค้า (เช่น "Junior Test Item") และราคา (เช่น `25`) จากนั้นคลิก **Create Product (สร้างสินค้า)**
8. **ผลลัพธ์ที่คาดหวัง**: มีข้อความแจ้งเตือนความสำเร็จ (Toast) แสดงขึ้นมา, กล่องโต้ตอบปิดลง และสินค้าชื่อ "Junior Test Item" จะถูกเพิ่มเข้าไปในตะกร้าโดยอัตโนมัติด้วยราคา `25.00` บาท

#### สถานการณ์ทดสอบ B: การจำลองการสแกนจากเครื่องสแกน (Hardware Scan Simulation)

เพื่อเป็นการจำลองว่ามีเครื่องสแกนส่งค่า `device_id` เข้ามาโดยที่ไม่ต้องใช้ฮาร์ดแวร์บอร์ด ESP32 จริง ๆ ให้คุณรันคำสั่งนี้ในเทอร์มินัล (Terminal) ของคุณครับ:

```bash
curl -X POST http://localhost:3000/api/scan \
  -H "Content-Type: application/json" \
  -H "X-Device-Key: <your_device_key>" \
  -d '{"barcode": "88888888", "device_id": "test_scanner"}'

```

*(อย่าลืมเปลี่ยน `<your_device_key>` ให้เป็นค่าของ `DEVICE_KEY` ที่กำหนดไว้ในไฟล์ `.env` ของคุณด้วยนะ)*

1. **ผลลัพธ์ที่คาดหวัง**: เอนด์พอยต์ (Endpoint) จะต้องปล่อยเหตุการณ์การสแกนนี้ออกไปยังสตรีม SSE
2. ในแท็บเบราว์เซอร์ของหน้า `/cashier` ที่กำลังเปิดใช้งานอยู่ กล่องโต้ตอบแจ้งเตือนควรจะเปิดขึ้นมาโดยอัตโนมัติเพื่อถามว่าต้องการลงทะเบียนบาร์โค้ด `88888888` หรือไม่

## สรุป (Conclusion)

งานชิ้นนี้จะช่วยให้คุณมีความเข้าใจที่ชัดเจนยิ่งขึ้นเกี่ยวกับวิธีที่ฝั่งหน้าบ้าน (Frontend) โต้ตอบกับสตรีม SSE, วิธีการเชื่อมต่อคอมโพเนนต์เฉพาะงานอย่าง `NewProductDialog` รวมถึงเข้าใจการทำงานของระบบความปลอดภัยในการเข้าถึงเส้นทางหลังบ้าน (Backend Route Security) ครับ 👍

ค่อย ๆ ไล่ดูทิศทางการไหลของข้อมูล (Data Flow) ให้ดีก่อนเริ่มแก้ไขโค้ดนะครับ และเมื่อทำเสร็จเรียบร้อยแล้ว อย่าลืมรันคำสั่งตรวจไวยากรณ์โค้ดและเช็กประเภทข้อมูลด้วยนะ:

```bash
npm run lint
```
